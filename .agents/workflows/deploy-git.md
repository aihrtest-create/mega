# Выполнение автоматического деплоя (GitHub → GitLab Mirror → Webhook)

Когда пользователь просит «деплой на гит» или запускает этот воркфлоу:

1. Проверь `git status` — убедись, что в коммит не попадёт ничего лишнего (временные файлы, `server/.env`, большие бинарники).
2. `git add .` (или адресно — только нужные файлы).
3. `git commit -m "<осмысленное сообщение>"`.
4. `git push origin main`.
5. **Проверь, что деплой реально доехал** (см. ниже) — не считай деплой успешным только по факту пуша.

## Как проверить, что деплой доехал до прода

Прод: https://party.hello-park.io/mega/

```bash
curl -s https://party.hello-park.io/mega/ | grep -oE 'index-[A-Za-z0-9_-]+\.js'
curl -sI https://party.hello-park.io/mega/ | grep -i last-modified
```

- Хэш JS-бандла (`index-*.js`) должен смениться, если менялись исходники фронтенда.
- `last-modified` = время сборки на сервере, должно стать свежим.
- Обычное время полного цикла: **~3 минуты** после пуша. Если коммит менял `package.json`/`package-lock.json` — дольше (сбрасывается кэш npm install в докере).

## Важные особенности

- Цепочка: push в GitHub → GitHub Action «Mirror to GitLab» → GitLab webhook → сервер выполняет `/opt/mega-deploy.sh` (`git reset --hard origin/main` + `docker compose up -d --build`).
- Коммит, **не меняющий исходники** (только доки, CI-файлы), не изменит сайт — докер возьмёт сборку из кэша. Это нормально, не ошибка.
- Фронтенд собирается **внутри** докер-образа; папка `dist/` не отслеживается git и на деплой не влияет.
- Историю на GitHub **нельзя переписывать** (rebase/force-push): зеркалирование идёт без `--force`, а сервер делает `reset --hard` — разъехавшаяся история ломает цепочку.
- Если через 10+ минут сайт не обновился: проверь статус Action «Mirror to GitLab» на GitHub и Recent events вебхука в GitLab (Settings → Webhooks). Если и то и другое успешно — проблема на сервере (место на диске, упавшая сборка), нужен доступ IT.
- Деплой можно перезапустить без пуша: GitLab → Settings → Webhooks → Test → Push events.
