# Интеграция AmoCRM ↔ Конфигуратор Hello Park

## Что это и зачем

Клиент оставляет заявку через Marquiz → лид появляется в AmoCRM. Менеджер переводит лид на этап «Отправить конфигуратор» → SalesBot дёргает вебхук нашего сервера → сервер генерирует **короткую персональную ссылку** и кладёт её примечанием в ленту карточки → менеджер отправляет ссылку клиенту → клиент заполняет конфигуратор → в AmoCRM появляется примечание с полным выбором клиента + обновляются поля сделки (дата, гости, сумма).

Всё работает на корпоративном сервере `party.hello-park.io` (фронтенд + backend + Redis в docker). Старый личный VDS (194.87.118.33) выведен из эксплуатации в июле 2026.

---

## Полная схема работы

```
1. Marquiz → клиент заполняет квиз на сайте
2. Marquiz → автоматически создаёт лид в AmoCRM (имя, телефон, ответы на квиз)
3. Менеджер открывает лид → переводит в этап «Отправить конфигуратор»
4. SalesBot (триггер: переход в этап):
   а) нода «Отправить webhook» → POST https://party.hello-park.io/api/amo/start-config
   б) нода «Внутреннее сообщение» → подсказка менеджеру посмотреть в ленту
5. Сервер по вебхуку:
   а) генерирует подписанную ссылку ?lead=<id>&sig=<hmac>
   б) сокращает её через Redis (nanoid, ключ invite:<id>, TTL 60 дней)
   в) добавляет примечание в карточку: «🔗 Персональная ссылка на конфигуратор: https://party.hello-park.io/mega/?id=Ab3x9Z»
6. Менеджер копирует ссылку из ленты и отправляет клиенту (Wazzup/WhatsApp)
7. Клиент открывает ссылку → фронтенд резолвит короткий id через GET /api/shorten
   и незаметно подменяет URL на полный ?lead=...&sig=... (history.replaceState)
8. Конфигуратор дёргает GET /api/leads/<id> → подтягивает имя и телефон из AmoCRM,
   в ленту падает примечание «👀 Клиент открыл конфигуратор»
9. Клиент проходит шаги → «Отправить» → POST /api/leads/<id>/configure
10. Сервер → примечание с полным выбором + обновление полей сделки
```

### Почему вебхук, а не ссылка в сообщении бота

- Нода SalesBot «Отправить webhook» умеет только слать запрос — прочитать ответ и подставить его в сообщение она не может. Поэтому ссылку в карточку кладёт сам сервер.
- Статичная ссылка вида `...?lead={{lead.id}}` **не сработает**: сервер принимает лидов только по подписанным ссылкам (HMAC-параметр `sig`), а подпись умеет генерировать только он сам.
- «Свой шаг (код)» с `widget_request` требует асинхронного протокола с `return_url` и работает только из шага виджета — сознательно не используем.

## Безопасность ссылок

- Ссылка на лид всегда содержит `sig` — первые 16 символов HMAC-SHA256 от lead_id с секретом `LEAD_SIGN_SECRET` (env; есть дефолт в `server/src/config/amo-config.js`).
- Все эндпоинты `/api/leads/:id` для числовых (Amo) id проверяют подпись — без валидного `sig` вернётся 403. Перебрать чужие лиды по номерам нельзя.
- Короткая ссылка `?id=Ab3x9Z` хранит внутри (в Redis) полный набор `leadId + sig`, поэтому для клиента она выглядит аккуратно, а работает как подписанная.
- Если Redis недоступен, сервер отдаёт длинную подписанную ссылку — интеграция деградирует, но не ломается.

## Как отличить Amo-лид от локального

Amo-лиды используют **числовой id** AmoCRM напрямую (`33153603`).
Локальные лиды (Telegram/Max бот) используют **nanoid** (`AzVBzGKakELv`).

Функция в `server/src/config/amo-config.js`:
```js
export function isAmoLeadId(id) {
  return typeof id === 'string' && /^\d+$/.test(id);
}
```

Сервер автоматически маршрутизирует запросы к нужному источнику.

```
├── src/
│   ├── services/
│   │   ├── amo.js                ← HTTP-клиент к AmoCRM API (getLead, getContact, addNoteToLead, updateLead...)
│   │   ├── amo-format.js         ← buildAmoNoteText() + buildAmoFieldUpdates()
│   │   └── redis.js              ← подключение к Redis (короткие ссылки)
│   ├── config/
│   │   └── amo-config.js         ← константы pipeline/stage/field IDs, isAmoLeadId(), signLeadId(), buildConfiguratorUrl()
│   └── routes/
│       ├── amo.js                ← POST /api/amo/start-config (вебхук SalesBot)
│       ├── shorten.js            ← GET /api/shorten?id=... (резолв коротких ссылок)
│       └── leads.js              ← GET/POST /api/leads/:id — Amo-id и nanoid, проверка sig
```

Фронтенд: `src/main.tsx` → компонент `ShortLinkResolver` перехватывает `?id=...`, резолвит через `/api/shorten` и подменяет URL на `?lead=...&sig=...`. Старые длинные ссылки работают без изменений.

## Эндпоинты сервера

### `POST /api/amo/start-config`
Вебхук для SalesBot. Принимает lead_id в любом привычном месте тела (JSON `{lead_id}` или form-encoded `leads[status][0][id]` — как шлёт Amo).

- Генерирует короткую подписанную ссылку (Redis, TTL 60 дней)
- **Добавляет примечание в карточку** со ссылкой — это основной канал доставки
- Отвечает `{ url, name, lead_id }` (на случай интеграций, умеющих читать ответ)

### `GET /api/shorten?id=<shortId>`
Резолв короткой ссылки → `{ leadId, sig, originalUrl }`. 404 если истекла или не существует.

### `GET /api/leads/:amoLeadId?sig=...`
Конфигуратор вызывает при загрузке, чтобы получить имя и телефон. Требует валидный `sig`.

### `POST /api/leads/:amoLeadId/opened?sig=...`
Клиент открыл конфигуратор → примечание «👀 Клиент открыл конфигуратор».

### `POST /api/leads/:amoLeadId/configure?sig=...`
Клиент нажал «Отправить» → примечание с полным выбором + PATCH полей сделки.

## Настройка SalesBot (если пересоздавать)

1. AmoCRM → Настройки → Чаты и мессенджеры → Список Salesbot → Создать
2. **Триггер:** «Переход в этап» → Мега Заявки → Отправить конфигуратор
3. **Шаг 1 — нода «Отправить webhook»:**
   ```
   https://party.hello-park.io/api/amo/start-config
   ```
   Больше в ней ничего заполнять не нужно.
4. **Шаг 2 — нода «Внутреннее сообщение»** (менеджерам):
   ```
   Персональная ссылка на конфигуратор добавлена в ленту карточки (примечание 🔗) — скопируй её и отправь клиенту.
   ```
   (опциональна — ссылка в ленте появится в любом случае)
5. Сохранить, убедиться что бот **включён**

## Обслуживание

### Когда токен истечёт (1 января 2028)
1. AmoCRM → amoМаркет → Установленные → карточка интеграции → Ключи и доступы
2. «Сгенерировать токен» (максимальный срок) → скопировать сразу
3. Передать IT: обновить `AMO_LONG_TOKEN` в `/opt/mega/server/.env` на корпоративном сервере
4. IT: `cd /opt/mega && docker compose up -d --force-recreate backend`
   (просто `restart` новые env-переменные НЕ подхватит)
5. Обновить токен и в локальном `server/.env`

### Добавить интеграцию для другой воронки (Ривьера, Сочи и т.д.)
1. Добавить этап «Отправить конфигуратор» в нужную воронку (UI: Настройки → Воронки)
2. Создать SalesBot с триггером на новый этап (см. выше) — вебхук тот же
3. Код сервера менять **не нужно** — он работает с любым числовым Amo-лид-ID

### Деплой изменений сервера
Пуш в GitHub `main` → зеркало в GitLab → вебхук → корпоративный сервер сам пересобирает docker (~1–3 мин). Прямого доступа к серверу нет — только через IT.

### Проверить что интеграция работает
```bash
curl -s -X POST https://party.hello-park.io/api/amo/start-config \
  -H "Content-Type: application/json" \
  -d '{"lead_id": 33535997}'
# Должен вернуть: {"url":"https://party.hello-park.io/mega/?id=...","name":"Тест 7","lead_id":"33535997"}
# name:null = проблема с Amo-токеном на сервере (ошибки Amo глотаются — смотреть логи backend)
# длинная ссылка ?lead=...&sig=... вместо короткой = проблема с Redis
```
Лид `33535997` («Тест 7») — тестовый, на нём можно гонять проверки. После вызова в его ленте должно появиться примечание со ссылкой.

## Ограничения и что можно улучшить

| Что | Статус |
|---|---|
| Работает только в Мега (SalesBot) | Легко добавить другие воронки — см. выше |
| Токен действует до 01.01.2028 | Перевыпускать вручную (напоминание — декабрь 2027) |
| `/api/amo/start-config` открыт без авторизации | Может создавать примечания в существующих лидах по номеру. Добавить `X-Api-Secret` заголовок (5 минут работы) |
| Повторные открытия конфигуратора = повторные примечания «Клиент открыл» | Добавить дедупликацию при необходимости |
| Поле «Пакет» (select, id=1374451) не заполняется | Нужно получить enum-ID вариантов через API и сопоставить |
| Короткие ссылки живут 60 дней | После истечения клиент увидит «Ссылка не найдена»; менеджер может перевыпустить, повторно прогнав этап |
